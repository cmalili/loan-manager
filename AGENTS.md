# AGENTS.md

## Project Overview
This repository is for **Loan Manager**, an internal loan management system for tracking:
- borrowers
- loans
- repayment schedules
- payments
- late charges
- audit history

This is a **serious software engineering project**, not a toy CRUD app. Prioritize correctness, auditability, and maintainability over speed or flashy features.

## Product Scope
Version 1 is an **internal tool** for staff/admin use.

Core capabilities:
- manage borrowers
- create and manage loans
- generate repayment schedules
- record payments
- apply payment allocations
- track overdue installments
- create late charges
- maintain audit logs

Out of scope unless explicitly requested:
- public borrower portal
- mobile app
- payment gateway integration
- AI underwriting / credit scoring
- multi-tenant architecture
- multi-currency support

## Technology Expectations
Backend stack:
- FastAPI
- SQLAlchemy ORM
- Alembic
- PostgreSQL
- Pydantic

Assume PostgreSQL is the source-of-truth database for persistence and migrations.

## Core Financial Rules
These are critical and must not be changed casually.

- Only **one active loan per borrower** in Version 1.
- Interest is **periodic interest**, not flat contract interest.
- Monthly rate: **30%**
- Weekly rate: **8%**
- Grace period: **7 days**
- Late charge: **one-time 10%** of the unpaid installment amount after the grace period
- Late charges are treated as a **separate financial record**
- Payment allocation order:
  1. late-charge interest
  2. late-charge principal
  3. regular interest
  4. regular principal
- Overpayments are **not allowed**
- Waived amounts must be tracked separately from paid amounts
- Backdated payments are allowed for admin users and must remain auditable

If a requested change conflicts with these rules, do not silently implement it. Call it out clearly.

## Money and Data Integrity
Financial correctness is more important than convenience.

- **Never use float for money**
- Use `Decimal` in Python and `NUMERIC(...)` in PostgreSQL
- Do not introduce approximate monetary math
- Do not silently discard or round away financial values without an explicit rule
- Do not delete financial history casually
- Prefer append-oriented history and auditability

Avoid storing redundant balance fields unless there is a clear reason and consistency is guaranteed.

## Architecture Rules
Follow separation of concerns.

- `models/` = SQLAlchemy ORM models
- `schemas/` = Pydantic request/response models
- `services/` = business logic
- `api/` = route handlers only
- `db/` = database setup/session/base
- `core/` = config/shared infrastructure

Important:
- keep **business logic out of route handlers**
- keep **database persistence details out of API schemas**
- do not place repayment or payment-allocation logic directly in FastAPI endpoints

## Preferred Implementation Style
When adding features:
1. understand the relevant business rule first
2. update or inspect models/schemas/services as needed
3. keep changes small and coherent
4. preserve migrations and data integrity
5. explain assumptions when they matter

Prefer:
- explicitness over cleverness
- readable code over compact code
- small functions with clear names
- consistent naming across models, schemas, and services

## Migrations and Schema Changes
Schema changes must be handled carefully.

- If models change, check whether Alembic migration changes are needed
- Do not hand-wave database changes
- Do not drop columns/tables or weaken constraints without clearly stating the impact
- Review autogenerated migrations before applying them
- Preserve important indexes, foreign keys, unique constraints, and check constraints

If a task implies a schema change, mention it explicitly.

## Testing Expectations
For logic involving money or state transitions, tests are expected.

High-priority test areas:
- loan creation constraints
- one-active-loan-per-borrower rule
- repayment schedule generation
- payment allocation ordering
- overpayment rejection
- overdue detection
- late-charge creation
- waived amount handling
- status transitions

Do not claim something is correct without checking where practical.

## Working Style for Codex
When working in this repo:

- inspect existing files before inventing new structure
- prefer modifying the existing architecture over introducing parallel patterns
- do not create unnecessary abstractions
- do not rename files or move modules unless there is a strong reason
- do not add dependencies unless needed
- if you add a dependency, explain why

When making edits:
- keep diffs focused
- avoid unrelated refactors
- preserve comments and intentional naming where possible

## Safety / Caution Areas
Be extra careful with:
- payment allocation logic
- audit history
- migration files
- auth / permissions
- schema constraints
- destructive operations

Never make destructive changes to financial data handling without clearly surfacing the risk.

## Definition of Done
A task is not done just because code was written.

A change is closer to done when:
- it matches the business rules
- it fits the repo structure
- schema impacts are handled
- migrations are considered
- obvious edge cases are addressed
- tests are added or updated when appropriate
- the result is understandable to a future maintainer

## If Uncertain
If requirements are ambiguous:
- prefer the documented business rules
- preserve existing constraints
- avoid making silent product decisions
- leave clear notes about assumptions
